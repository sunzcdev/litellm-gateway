# LiteLLM Gateway — 软件需求说明书

## 1. 引言

### 1.1 目的

本文档定义 LiteLLM Gateway 的详细软件需求，作为实现和验收的依据。

### 1.2 范围

系统包含以下核心模块：
- LiteLLM Proxy 服务（核心路由网关）
- 免费模型自动探索器（auto-discover）
- 配置热加载与变更管理
- 日志与用量统计
- 部署与运维

### 1.3 定义与缩写

| 术语 | 说明 |
|------|------|
| LiteLLM | 开源 LLM 网关，提供 100+ 供应商统一接入 |
| Proxy | LiteLLM 的 HTTP 代理服务形态 |
| cost-based routing | 基于成本的路由策略，自动选最低价模型 |
| fallback | 首选模型失败后的自动降级切换 |
| SIGHUP | Unix 信号，LiteLLM 监听此信号重载配置 |
| config-only | 不使用数据库，配置全部写在 YAML 文件中 |

### 2.2 部署拓扑

```
进程: litellm --config config.yaml --port 4000
模式: config-only（无 Postgres）
日志: SQLite → ~/.litellm/litellm.db
探索: cron 每日运行 discover.py
```

## 3. 功能需求

### FR-01：模型别名路由

| 项目 | 说明 |
|------|------|
| 优先级 | P0 |
| 描述 | 客户端传入 `model=smart`，LiteLLM 根据策略选择实际模型 |
| 输入 | `model_name: smart` |
| 输出 | 路由到成本最低/优先级最高的可用模型 |
| 验证 | 连续调用 10 次 `model=smart`，应命中配置中的多个后端模型（非同一家） |

### FR-02：成本优先路由

| 项目 | 说明 |
|------|------|
| 优先级 | P0 |
| 描述 | 同一别名下的多个模型，按成本排序，优先选 cost=0 的 |
| 策略 | `routing_strategy: cost-based-routing` |
| 验证 | `smart` 别名下有 OpenRouter:free 和 DeepSeek 直连，应永远走 free |

### FR-03：Fallback 链

| 项目 | 说明 |
|------|------|
| 优先级 | P0 |
| 描述 | 首选模型 429/500/超时 时自动切到下一个 |
| 验证 | 屏蔽首选模型端口后，请求应在 2s 内 fallback 到次选 |

### FR-04：免费模型自动探索

| 项目 | 说明 |
|------|------|
| 优先级 | P0 |
| 描述 | 每日自动扫描各平台免费模型，更新 config |
| 扫描范围 | OpenRouter, Groq, Gemini AI Studio |
| 对比策略 | 与本地缓存比较，检测：新增/消失/价格变动 |
| 输出 | config.yaml 更新 + SIGHUP 热加载 + 通知 |
| 验证 | 手动 mock 一个新免费模型，discover.py 应检测到并写入 config |

### FR-05：模型转付费告警

| 项目 | 说明 |
|------|------|
| 优先级 | P1 |
| 描述 | 之前 free 的模型变成付费 → 自动摘除 + 通知 |
| 通知方式 | 写入日志文件 + 可选 webhook |

### FR-06：零付费预算保护

| 项目 | 说明 |
|------|------|
| 优先级 | P0 |
| 描述 | 任何请求的超额费用自动拦截 |
| 配置 | `max_budget: 0`, `budget_duration: 1d` |
| 验证 | 用测试模型名触发付费调用，应返回 429 budget exceeded |

### FR-07：日志记录

| 项目 | 说明 |
|------|------|
| 优先级 | P0 |
| 描述 | 每次调用记录关键信息到 SQLite |
| 字段 | 时间、请求模型、实际模型、供应商、input tokens、output tokens、耗时、状态、错误 |
| 验证 | 调用 5 次后查询日志表，应有 5 条记录 |

### FR-08：用量统计

| 项目 | 说明 |
|------|------|
| 优先级 | P1 |
| 描述 | 通过 LiteLLM 内置 UI `/ui` 查看用量 |
| 指标 | 请求量、Token 数、耗时分布、失败率 |

### FR-09：配置热加载

| 项目 | 说明 |
|------|------|
| 优先级 | P0 |
| 描述 | discover.py 更新 config 后，LiteLLM 自动重载 |
| 方式 | kill -SIGHUP $(pidof litellm) 或 LiteLLM API |
| 验证 | 修改 config 后发 SIGHUP，再调用 `/v1/models` 应看到变化 |

### FR-10：多协议支持

| 项目 | 说明 |
|------|------|
| 优先级 | P1 |
| 描述 | 同时支持 OpenAI 和 Anthropic 协议 |
| OpenAI | `/v1/chat/completions` |
| Anthropic | `/v1/messages` |
| 验证 | 用 Anthropic SDK 发请求到 LiteLLM，应能正常路由 |

### FR-11：本地模型集成

| 项目 | 说明 |
|------|------|
| 优先级 | P1 |
| 描述 | 对接本地 Ollama，Qwen2.5-1.5B 加入路由池 |
| 协议 | `ollama/` 前缀 |
| 配置 | `api_base: http://localhost:11434` |

### FR-12：Key 保护

| 项目 | 说明 |
|------|------|
| 优先级 | P1 |
| 描述 | 真实 API Key 仅配置在服务器端 |
| 实现 | .env 文件独立管理，不进入 config.yaml |

## 4. 非功能需求

### NFR-01：性能

| 项目 | 指标 |
|------|------|
| 额外延迟 | < 50ms（个人场景远低于 LiteLLM 8ms 基准） |
| 并发 | > 10 QPS 即可（个人使用） |
| 内存占用 | < 300MB |

### NFR-02：可用性

| 项目 | 指标 |
|------|------|
| 自愈 | 进程崩溃后自动重启（systemd/supervisor） |
| 热加载 | 配置更新不中断服务 |

### NFR-03：安全性

| 项目 | 指标 |
|------|------|
| 监听范围 | 127.0.0.1 仅本地 |
| Key 存储 | .env 文件，不进 git |
| 预算保护 | `max_budget: 0` 硬屏障 |

### NFR-04：可维护性

| 项目 | 指标 |
|------|------|
| 启动时间 | < 10s |
| 日志轮转 | 自动清理 30 天前的日志 |
| 配置复杂度 | 单 YAML 文件，注释完整 |

## 5. 接口规范

### 5.1 调用方接口（对外）

```
POST /v1/chat/completions
Content-Type: application/json
Authorization: Bearer <LITELLM_MASTER_KEY>

{
  "model": "smart",
  "messages": [{"role": "user", "content": "你好"}]
}
```

### 5.2 管理接口（对内）

| 路径 | 说明 |
|------|------|
| `/v1/models` | 模型列表 |
| `/health/liveliness` | 存活检查 |
| `/health/readiness` | 就绪检查 |
| `/ui` | 管理界面 |
| `/spend/logs` | 调用日志 |

## 6. 验收标准

| 序号 | 场景 | 预期结果 |
|------|------|---------|
| AC-1 | 调用 `model=smart` | 成功返回结果，路由到免费模型 |
| AC-2 | 屏蔽首选模型后调用 | 3s 内 fallback 到次选 |
| AC-3 | 运行 discover.py | 扫描各平台，写入新模型 |
| AC-4 | 配置付费模型后调用 | 返回 429 budget exceeded |
| AC-5 | 查询 `/ui` | 显示调用日志和用量统计 |
| AC-6 | SIGHUP 后查询模型列表 | 新模型可见 |
| AC-7 | 本地 Ollama 加入路由池 | `model=local` 返回本地模型结果 |
